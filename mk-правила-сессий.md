# Система сессий в Claude Code

> Как не терять контекст между разговорами и держать историю каждой задачи.
> Памятка для участников МК «Вайбкодинг Техспеца». Бери и внедряй у себя.

---

## Зачем это вообще

У Claude Code короткая память: закрыл чат — и весь контекст по задаче испарился. Через неделю ты сам не вспомнишь, *почему* выбрал именно такой вариант, что уже пробовал и что не сработало.

**Сессия** решает это. Сессия — это папка с историей одной темы / задачи / вопроса. В ней лежит не только результат, но и **процесс**: как думали, какие варианты рассматривали, где свернули не туда и как выкрутились.

Одна тема = одна сессия. Сессия может длиться хоть несколько дней — ты возвращаешься к ней, а Claude поднимает всю историю и продолжает с того же места.

**Что это даёт:**
- Открыл новый чат — Claude видит твои незакрытые задачи и предлагает продолжить.
- Через месяц можешь восстановить логику любого решения.
- Не повторяешь одни и те же ошибки — они записаны.
- Несколько задач параллельно не мешаются друг другу.

---

## Как это работает в двух словах

1. Один раз кладёшь правила в свой `CLAUDE.md` (блок ниже — скопировать и вставить).
2. После этого **в начале каждого разговора** Claude сам проверяет папку с сессиями и предлагает: продолжить открытую или завести новую.
3. По ходу работы Claude пишет лог и фиксирует решения. Тебе ничего вручную вести не надо — он делает это сам.

---

## Структура папок

В корне рабочего проекта одна папка:

```
sessions/
├── active/      # текущие сессии
├── archive/     # завершённые
└── _index.md    # оглавление всех сессий
```

Если папки нет — Claude создаёт её при первой задаче.

---

## Как устроена одна сессия

**Папка:** `sessions/active/YYYY-MM-DD_NNN_краткое-название/`

- `YYYY-MM-DD` — дата начала
- `NNN` — порядковый номер (001, 002, 003…), берётся следующий по `_index.md`
- `краткое-название` — 2–4 слова через дефис, суть задачи

Пример: `2026-06-27_015_лендинг-под-курс/`

Внутри папки — **3 файла**.

### 1. `session.md` — паспорт сессии

```markdown
# Сессия NNN: Название

- **UUID:** (генерится командой `uuidgen`)
- **Дата начала:** YYYY-MM-DD
- **Модуль:** (тег области: development, content, design, research, business, personal…)
- **Статус:** active

## Резюме
Краткое описание задачи в 1–3 предложения. Обновляется по ходу.

## Связанные сессии
- (ссылки на пересекающиеся сессии, если есть)
```

### 2. `log.md` — хронология работы

```markdown
# Лог сессии NNN

## YYYY-MM-DD

### Задача
Что попросил — своими словами.

### Процесс
1. Что сделали первым шагом и почему
2. Какие варианты рассматривали
3. Сколько итераций, что правили между ними
4. Что пошло не так и как исправили

### Результат
- Что получилось, где лежит файл/код
- Неочевидные решения и развилки

### Расход
- Оценка: лёгкая / средняя / тяжёлая задача
```

> Лог — это не «что сделано», а **как делали**. Чтобы потом восстановить весь ход мысли.

### 3. `memory.md` — факты и решения

```markdown
# Memory сессии NNN

## Ключевые решения
- Решение 1 и почему его приняли
- Решение 2

## Важные факты
- То, что пригодится при возвращении к задаче
```

> Сюда — то, что **нельзя вывести из кода**: контекст, причины решений, ограничения, договорённости.

## Многодневные сессии

Дата в названии папки **не меняется** — она фиксирует начало. Каждый новый день — просто новый блок в `log.md`:

```markdown
## 2026-06-27

### Задача
...

## 2026-06-28

### Продолжение
...
```

---

## Оглавление — `_index.md`

При создании и закрытии сессии обновляется общий индекс:

```markdown
# Индекс сессий

## Активные

| #   | Дата       | Название          | Модуль      | Краткое содержание           |
|-----|------------|-------------------|-------------|------------------------------|
| 001 | 2026-06-27 | настройка-проекта | development  | Линтеры, CI, первый деплой    |

## Архив

| #   | Дата       | Название | Содержание |
|-----|------------|----------|------------|
```

---

## Связанные сессии

Если две сессии пересекаются по теме — связывай их через поле `## Связанные сессии` в `session.md`.

Связь **всегда двусторонняя**: если сессия A ссылается на B, то и B должна ссылаться на A.

```markdown
## Связанные сессии
- UUID → NNN название — почему связаны
```

---

## Закрытие и архивация

- Закрывается сессия **только по твоей команде**: «закрой сессию», «архивируй», «готово».
- При закрытии: папка переезжает из `active/` в `archive/`, обновляется `_index.md`.
- Если в `active/` накопилось больше 7 сессий — Claude сам предложит заархивировать старые.

Сама по себе сессия не закрывается — пока не сказал, она «живая» и ждёт продолжения.

---

## Имя чата в истории Claude Code

Чтобы потом находить нужный разговор, называй чаты единообразно.

**Формат:** `Тема [NNN | модуль] DD.MM.YY`

**Примеры:**
- `Настройка CI/CD [001 | development] 27.06.26`
- `Редизайн лендинга [002 | design] 27.06.26`

После создания сессии Claude подскажет, как назвать чат.

## ⚙️ Как включить у себя (ОБЯЗАТЕЛЬНЫЙ шаг)

Без этого Claude не будет вести сессии сам. Нужно один раз положить правила в `CLAUDE.md`.

**Куда:**
- Глобально для всех проектов → `~/.claude/CLAUDE.md`
- Только для одного проекта → `CLAUDE.md` в корне проекта

**Что вставить** — скопируй блок целиком:

```markdown
## Система сессий

Каждая тема / задача / вопрос = отдельная сессия. Сессия хранит историю работы,
процесс и решения. Сессия может длиться несколько дней.

### В начале КАЖДОГО разговора

1. Проверь: `ls sessions/active/` в текущем проекте
2. Если папки `sessions/` нет — создай структуру:
   sessions/active/, sessions/archive/, sessions/_index.md
3. Если есть открытые сессии — покажи список, предложи продолжить подходящую
4. Если подходящей нет — спроси тему и создай новую сессию
5. Никогда не начинай работу без сессии — сначала сессия, потом работа

### Создание сессии

Папка: sessions/active/YYYY-MM-DD_NNN_краткое-название/
Внутри — 3 файла: session.md (паспорт), log.md (хронология «как делали»),
memory.md (решения и факты, которые нельзя вывести из кода).

- UUID для session.md — генерируй командой `uuidgen`.
- NNN — следующий порядковый номер по _index.md.

### Многодневные сессии

Дата в названии папки не меняется. Каждый новый день — новый блок в log.md.

### Индекс

При создании/закрытии сессии обновляй sessions/_index.md (таблицы Активные / Архив).

### Закрытие

Только по команде пользователя («закрой / архивируй / готово»):
перенеси папку из active/ в archive/, обнови _index.md.
Если в active/ больше 7 сессий — предложи заархивировать старые.

### Связанные сессии

Если сессии пересекаются по теме — связывай через поле в session.md.
Связь всегда двусторонняя.

### Имя чата

После создания сессии предложи формат: `Тема [NNN | модуль] DD.MM.YY`.
```

> Полную, расширенную версию этого блока (с готовыми шаблонами всех трёх файлов) можно взять из этой памятки выше — раздел «Как устроена одна сессия».

## Частые вопросы

**Обязательно вести все 3 файла?**
Да — это минимум. `session.md` = что за задача, `log.md` = как шли, `memory.md` = почему так решили. Claude заполняет их сам, тебе достаточно работать как обычно.

**А если задача на 5 минут?**
Всё равно сессия. Мелкие пишутся быстро, зато потом всё в одном месте и ничего не теряется.

**Сколько сессий держать в `active/`?**
До ~7. Больше — заархивируй закрытые, чтобы список не зарастал.

**Зачем UUID, если есть номер?**
Номер можно случайно повторить или сдвинуть. UUID — железобетонная привязка для ссылок между сессиями.

**Что писать в `log.md`, а что в `memory.md`?**
Лог — хронология действий («сделал → проверил → переделал»). Memory — выжимка решений и контекста, которую не достать из кода («выбрали X, потому что Y»).
